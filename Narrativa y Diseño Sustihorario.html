1. Introducción

Contextualización del Problema

En el entorno educativo actual, la gestión eficiente de los recursos humanos es crítica para mantener la calidad de la enseñanza. Específicamente, en la educación secundaria y superior, las ausencias imprevistas del profesorado generan disrupciones significativas en el horario lectivo. La gestión de estas ausencias, conocida como "guardias" o "sustituciones", representa un desafío administrativo constante.

Tradicionalmente, muchos centros educativos realizan esta gestión de manera manual o semimanual, dependiendo de hojas de cálculo desconectadas, tablones de anuncios físicos, cadenas de correos electrónicos o grupos de mensajería instantánea. Este enfoque fragmentado conlleva múltiples problemáticas:

  • Falta de transparencia: El reparto de guardias suele percibirse como arbitrario o injusto.
  • Sobrecarga docente: La ausencia de un control centralizado provoca que ciertos profesores asuman una carga desproporcionada de sustituciones.
  • Errores operativos: La gestión manual es propensa a fallos, como asignar a profesores no disponibles o dejar aulas sin supervisión.
  • Ineficiencia temporal: Los coordinadores invierten horas valiosas en resolver rompecabezas logísticos cada mañana.

Propuesta de Valor: SustiHorario

SustiHorario nace como respuesta directa a estas ineficiencias. Se trata de una solución tecnológica integral diseñada para digitalizar, automatizar y optimizar el flujo completo de gestión de sustituciones. Su propuesta de valor se fundamenta en tres pilares:

  1. Automatización Inteligente: Algoritmos que sugieren asignaciones basadas en criterios objetivos de disponibilidad y equidad.
  2. Accesibilidad Total: Una aplicación móvil nativa que permite a los docentes consultar y gestionar su estado desde cualquier lugar.
  3. Transparencia y Equidad: Un sistema de registro auditable que garantiza un reparto justo de la carga laboral.
Objetivo General: Desarrollar una herramienta digital robusta y escalable que transforme la gestión de sustituciones de un problema administrativo diario a un proceso automatizado y transparente.

2. Estado del Arte

Análisis de Soluciones Existentes

El mercado actual de software educativo ofrece soluciones potentes pero a menudo sobredimensionadas o poco accesibles para centros pequeños y medianos. Las referencias principales analizadas son:

Solución Características Principales Limitaciones Detectadas
WebUntis Plataforma integral de gestión académica, planificación de horarios y comunicación. Coste elevado, complejidad de implementación y curva de aprendizaje pronunciada para funcionalidades simples.

Innovación Tecnológica

SustiHorario se diferencia por su enfoque tecnológico moderno y ágil. A diferencia de soluciones legacy basadas en web tradicionales o aplicaciones de escritorio pesadas, SustiHorario apuesta por:

  • Flutter & Dart: Para ofrecer una experiencia nativa fluida tanto en iOS como en Android con una única base de código, reduciendo tiempos de desarrollo y mantenimiento.
  • Firebase (Backend-as-a-Service): Permite una arquitectura serverless que garantiza sincronización en tiempo real, autenticación segura y escalabilidad automática sin costes de infraestructura iniciales.

Ventaja Competitiva de SustiHorario

Diferenciadores clave frente a la competencia:

4. Análisis de Requisitos

El sistema se ha diseñado atendiendo a las necesidades de tres perfiles de usuario diferenciados: Profesores, Coordinadores y Administradores.

Requisitos Funcionales Detallados

RF-01: Autenticación y Registro

  • RF-01.1: El sistema debe permitir login con email y contraseña mediante Firebase Authentication.
  • RF-01.2: El registro debe solicitar: nombre, email, contraseña, rol (profesor/coordinador/admin) y centro educativo.
  • RF-01.3: Redirección automática post-login según rol: profesores → ProfesoresHomePage, coordinadores → CoordinadorHomePage, admins → AdminPage.
  • RF-01.4: Validación de seguridad: contraseña mínimo 8 caracteres, email formato válido.

RF-02: Panel del Profesor

  • RF-02.1: Visualización de horario semanal en vista de cuadrícula (HorariosPage) con clases fijas asignadas.
  • RF-02.2: Marcado de disponibilidad para guardias por día/hora con interfaz visual intuitiva.
  • RF-02.3: Consulta de guardias asignadas en tiempo real (MisGuardiasPage) mostrando: fecha, hora, aula, asignatura, profesor ausente.
  • RF-02.4: Solicitud de bajas mediante formulario (SolicitarBajaPage): fecha inicio/fin, tipo (enfermedad, personal, formación), horas afectadas.
  • RF-02.5: Función de adición de clases fijas mediante _anadirClase() con validación de conflictos.

RF-03: Panel del Coordinador

  • RF-03.1: Dashboard centralizado (CoordinadorHomePage) con resumen de bajas pendientes y guardias del día.
  • RF-03.2: Gestión de bajas pendientes (BajasPendientesPage): aprobación/rechazo, visualización de horas afectadas.
  • RF-03.3: Asignación de guardias (GuardiasAsignadasPage):
    • Asignación manual seleccionando profesor disponible.
    • Sugerencia automática basada en disponibilidad y carga equilibrada.
    • Reasignación de guardias existentes.
  • RF-03.4: Creación y gestión de modelos de horario (ModelosHorarioPage, EditarModeloPage).
  • RF-03.5: Panel de notificaciones (NotificacionesPage) con alertas de sobrecarga o falta de cobertura.
  • RF-03.6: Gestión de usuarios del centro (UsuariosPage): visualización, eliminación de cuentas.
  • RF-03.7: Configuración de límites de guardias por profesor (ConfiguracionPage).

RF-04: Panel del Administrador

  • RF-04.1: Creación de centros educativos con datos: nombre, dirección, configuración global.
  • RF-04.2: Gestión centralizada de múltiples centros desde AdminPage.

RF-05: Persistencia de Datos

  • RF-05.1: Guardado automático en Firestore mediante _guardarHorarioEnFirestore() con merge: true.
  • RF-05.2: Sincronización en tiempo real de cambios en horarios y guardias.
  • RF-05.3: Timestamps automáticos (updatedAt) para auditoría.

5. Diseño

Arquitectura de Datos (NoSQL)

Dado el uso de Cloud Firestore, se ha optado por un modelo de datos documental flexible y optimizado para lecturas rápidas, evitando la rigidez de los esquemas relacionales tradicionales.

Colecciones Principales y Estructura de Datos:

Colección Document ID Campos Principales Tipo de Datos
centros Auto-generado id, nombre, direccion String
users uid (Firebase Auth) uid, nombre, email, rol, centroId String, String, String, String (enum), Reference
horarios uid (igual que user) uid, nombreProfesor, centroId, horarioFijo (Map), disponibilidad (Map), updatedAt String, String, Reference, Map<String, Map>, Timestamp
bajas Auto-generado id, profesorId, fechaInicio, fechaFin, horasAfectadas (Array), tipo, estado String, Reference, Timestamp, Timestamp, Array<String>, String, String
guardias Auto-generado id, dia, hora, profesorAusenteId, sustitutoId, asignatura, curso, aula, tipo String, String, String, Reference, Reference, String, String, String, String

Relaciones entre Entidades:

  • Centro → Usuarios (1:N): Un centro tiene múltiples profesores, coordinadores y administradores. Relación mediante campo centroId.
  • Usuario → Horario (1:1): Cada usuario tiene exactamente un documento de horario con el mismo uid.
  • Usuario → Bajas (1:N): Un profesor puede solicitar múltiples bajas. Relación mediante profesorId.
  • Guardia → Usuarios (N:2): Cada guardia referencia a dos usuarios: profesorAusenteId y sustitutoId.

Diseño Lógico: Denormalización para Performance

Firestore es NoSQL orientado a documentos, por lo que se aplica denormalización estratégica para optimizar lecturas (mucho más frecuentes que escrituras en este dominio):

  • Mapas embebidos: horarioFijo y disponibilidad se almacenan como mapas anidados dentro del documento de horario, evitando múltiples subconsultas.
  • Arrays para datos temporales: horasAfectadas en bajas se guarda como array de strings para acceso directo.
  • Referencias (Foreign Keys): Se usan campos como centroId, profesorId para mantener integridad referencial lógica sin joins.
  • Duplicación controlada: nombreProfesor en horarios evita consultas adicionales a users para mostrar vistas rápidas.

Equivalente SQL (Referencia Teórica)

Aunque SustiHorario no usa SQL, a continuación se presenta la traducción equivalente a modelo relacional para comprensión académica:

-- Tabla centros
CREATE TABLE centros (
  id VARCHAR(50) PRIMARY KEY,
  nombre VARCHAR(200) NOT NULL,
  direccion TEXT
);

-- Tabla usuarios
CREATE TABLE usuarios (
  uid VARCHAR(50) PRIMARY KEY,
  nombre VARCHAR(200) NOT NULL,
  email VARCHAR(200) UNIQUE NOT NULL,
  rol ENUM('profesor', 'coordinador', 'admin') NOT NULL,
  centro_id VARCHAR(50),
  FOREIGN KEY (centro_id) REFERENCES centros(id)
);

-- Tabla horarios
CREATE TABLE horarios (
  uid VARCHAR(50) PRIMARY KEY,
  nombre_profesor VARCHAR(200),
  centro_id VARCHAR(50),
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (uid) REFERENCES usuarios(uid),
  FOREIGN KEY (centro_id) REFERENCES centros(id)
);

-- Tabla horario_fijo (normalizada)
CREATE TABLE horario_fijo (
  id INT AUTO_INCREMENT PRIMARY KEY,
  horario_uid VARCHAR(50),
  hora TIME,
  dia ENUM('Lunes','Martes','Miércoles','Jueves','Viernes'),
  clase VARCHAR(200),
  FOREIGN KEY (horario_uid) REFERENCES horarios(uid)
);

-- Tabla disponibilidad (normalizada)
CREATE TABLE disponibilidad (
  id INT AUTO_INCREMENT PRIMARY KEY,
  horario_uid VARCHAR(50),
  hora TIME,
  dia ENUM('Lunes','Martes','Miércoles','Jueves','Viernes'),
  disponible BOOLEAN DEFAULT TRUE,
  FOREIGN KEY (horario_uid) REFERENCES horarios(uid)
);

-- Tabla bajas
CREATE TABLE bajas (
  id VARCHAR(50) PRIMARY KEY,
  profesor_uid VARCHAR(50),
  fecha_inicio DATE,
  fecha_fin DATE,
  tipo VARCHAR(50),
  estado VARCHAR(50),
  FOREIGN KEY (profesor_uid) REFERENCES usuarios(uid)
);

-- Tabla guardias
CREATE TABLE guardias (
  id VARCHAR(50) PRIMARY KEY,
  dia VARCHAR(20),
  hora TIME,
  profesor_ausente_uid VARCHAR(50),
  sustituto_uid VARCHAR(50),
  asignatura VARCHAR(100),
  curso VARCHAR(50),
  aula VARCHAR(50),
  tipo VARCHAR(50),
  FOREIGN KEY (profesor_ausente_uid) REFERENCES usuarios(uid),
  FOREIGN KEY (sustituto_uid) REFERENCES usuarios(uid)
);
Nota: Esta estructura SQL es solo conceptual. Firestore utiliza un modelo documental que prioriza lecturas rápidas sin necesidad de JOINs costosos, resultando en mejor rendimiento para consultas frecuentes de horarios.

Funciones Clave Implementadas

1. Función _guardarHorarioEnFirestore()

Ubicación: lib/pages/profesor/horarios_page.dart

Propósito: Persistir el estado completo del horario del profesor en Firestore con sincronización en tiempo real.

Future<void> _guardarHorarioEnFirestore() async {
  try {
    final user = FirebaseAuth.instance.currentUser;
    if (user == null) return;
    
    await FirebaseFirestore.instance
        .collection('horarios')
        .doc(user.uid)
        .set({
          'uid': user.uid,
          'nombreProfesor': user.displayName ?? 'Sin nombre',
          'centroId': _centroId,
          'horarioFijo': _horarioFijo,
          'disponibilidad': _disponibilidad,
          'updatedAt': FieldValue.serverTimestamp(),
        }, SetOptions(merge: true));
    
    ScaffoldMessenger.of(context).showSnackBar(
      SnackBar(content: Text('✓ Horario guardado correctamente')),
    );
  } catch (e) {
    ScaffoldMessenger.of(context).showSnackBar(
      SnackBar(content: Text('✗ Error al guardar: $e')),
    );
  }
}

Características:

  • Usa SetOptions(merge: true) para actualización parcial sin sobrescribir
  • Timestamp automático con FieldValue.serverTimestamp()
  • Manejo de errores con try-catch y feedback visual

2. Función _anadirClase()

Ubicación: lib/pages/profesor/horarios_page.dart

Propósito: Validar y añadir una clase al horario fijo del profesor evitando conflictos.

void _anadirClase(String hora, String dia, String asignatura) {
  // Validación de conflictos con disponibilidad
  if (_disponibilidad[hora]?[dia] == true) {
    ScaffoldMessenger.of(context).showSnackBar(
      SnackBar(content: Text('⚠️ Conflicto: Ya marcaste disponibilidad en ese horario')),
    );
    return;
  }
  
  // Añadir a horarioFijo
  setState(() {
    if (_horarioFijo[hora] == null) {
      _horarioFijo[hora] = {};
    }
    _horarioFijo[hora][dia] = asignatura;
  });
  
  // Persistir cambios
  _guardarHorarioEnFirestore();
}

Validaciones implementadas:

  • Detecta solapamientos con disponibilidad declarada
  • Inicializa estructuras de mapas anidados si no existen
  • Actualiza UI inmediatamente con setState()

3. Función _login()

Ubicación: lib/pages/loginYregister/login_page.dart

Propósito: Autenticar usuario y redirigir según rol.

Future<void> _login() async {
  if (!_formKey.currentState!.validate()) return;
  
  setState(() => _isLoading = true);
  
  try {
    // Autenticación Firebase
    final credential = await FirebaseAuth.instance.signInWithEmailAndPassword(
      email: _emailController.text.trim(),
      password: _passwordController.text,
    );
    
    // Obtener rol del usuario
    final userDoc = await FirebaseFirestore.instance
        .collection('users')
        .doc(credential.user!.uid)
        .get();
    
    final rol = userDoc.data()?['rol'] ?? 'profesor';
    
    // Redirección basada en rol
    if (rol == 'profesor') {
      context.go('/profesor-home');
    } else if (rol == 'coordinador') {
      context.go('/coordinador-home');
    } else if (rol == 'admin') {
      context.go('/admin');
    }
  } on FirebaseAuthException catch (e) {
    String message = 'Error de autenticación';
    if (e.code == 'user-not-found') {
      message = 'Usuario no encontrado';
    } else if (e.code == 'wrong-password') {
      message = 'Contraseña incorrecta';
    }
    ScaffoldMessenger.of(context).showSnackBar(
      SnackBar(content: Text(message)),
    );
  } finally {
    setState(() => _isLoading = false);
  }
}

Future<void> _registrar() async {

Seguridad y Calidad de Código

Se han implementado buenas prácticas de desarrollo para asegurar la robustez del aplicativo:

  • Firestore Security Rules: Lógica de validación en el lado del servidor para prevenir accesos no autorizados y corrupción de datos.
  • Manejo de Errores: Bloques try-catch exhaustivos y feedback visual al usuario mediante SnackBars.
  • Estructura Modular: Separación clara de responsabilidades (Model-View-ViewModel/Provider) para facilitar el mantenimiento.

Firestore Security Rules Implementadas

Archivo: firestore.rules

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Función auxiliar: usuario autenticado
    function isSignedIn() {
      return request.auth != null;
    }
    
    // Función auxiliar: es el mismo usuario
    function isOwner(uid) {
      return request.auth.uid == uid;
    }
    
    // Función auxiliar: tiene rol específico
    function hasRole(role) {
      return isSignedIn() && 
             get(/databases/$(database)/documents/users/$(request.auth.uid)).data.rol == role;
    }
    
    // Colección users: solo lectura propia, coordinadores/admins pueden listar
    match /users/{userId} {
      allow read: if isOwner(userId) || hasRole('coordinador') || hasRole('admin');
      allow write: if isOwner(userId) || hasRole('admin');
    }
    
    // Colección horarios: lectura propia y coordinadores, escritura solo propietario
    match /horarios/{horarioId} {
      allow read: if isOwner(horarioId) || hasRole('coordinador');
      allow write: if isOwner(horarioId);
    }
    
    // Colección bajas: crear propias, coordinadores gestionan todas
    match /bajas/{bajaId} {
      allow read: if isSignedIn();
      allow create: if isSignedIn() && request.resource.data.profesorId == request.auth.uid;
      allow update, delete: if hasRole('coordinador');
    }
    
    // Colección guardias: coordinadores gestionan
    match /guardias/{guardiaId} {
      allow read: if isSignedIn();
      allow write: if hasRole('coordinador');
    }
    
    // Colección centros: solo admins crean/editan, todos leen
    match /centros/{centroId} {
      allow read: if isSignedIn();
      allow write: if hasRole('admin');
    }
  }
}

Características de Seguridad:

  • Autenticación obligatoria: Todas las operaciones requieren request.auth != null
  • Autorización por roles: Funciones auxiliares verifican roles desde Firestore
  • Principio de mínimo privilegio: Cada usuario solo accede a lo estrictamente necesario
  • Validación server-side: Las reglas se ejecutan en el servidor, imposibles de bypass desde cliente

Patrones de Arquitectura Aplicados

1. Provider Pattern (State Management)

Gestión de estado reactivo con separación de lógica de negocio:

  • HorarioProvider: Centraliza estado de horarios, sincroniza con Firestore, notifica cambios a UI
  • UsuariosProvider: Gestiona lista de usuarios del centro, operaciones CRUD
  • Beneficio: UI reactiva automáticamente a cambios de datos, reducción de setState manual

2. Repository Pattern

Abstracción de la capa de datos:

  • Modelos (*_model.dart) con métodos fromJson() y toJson()
  • Lógica de acceso a Firestore encapsulada en Providers
  • Facilita testing con mocks y futura migración de backend

3. MVVM (Model-View-ViewModel)

  • Model: centro_model.dart, user_model.dart, horario_model.dart
  • View: Páginas en lib/pages/
  • ViewModel: Providers en lib/providers/
